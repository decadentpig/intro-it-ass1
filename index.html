Project Idea

Overview

My project idea is for a browser-based, asynchronous multiplayer factory management game, themed around a vaccine “arms race” - a Jabs-In-Arms race. Designed to be checked-in with two or three times a day, the game sees players tasked with creating an efficient production and distribution flow toward a final goal, while attempting to sabotage fellow players by various means such as bottlenecking supply, cutting power and dumping waste. In short, think Settlers of Catan meets Factorio, with the social aspect and general pace of fantasy football or Words with Friends. And all of it available on mobile or in your web browser in a sleek, minimalist representation.

Motivation

A game of this sort is interesting to me for a few reasons. Not least of all thematically, as vaccine development and rollout in the COVID era is likely to be a topic at the forefront of current affairs for some time to come. And as these events drag on, people will no doubt continue to search for new ways to interact remotely with their friends. Providing a platform for socially-distant interaction that also explores some subject matter is something I’m interested in applying myself to.

Additionally, mobile games are more popular than ever, with time spent playing them increasing 8-10% every year (Business Wire 2020, para. 3). Mobile games with deceptively simple gameplay loops and rewards for repetitive behaviour perform really well. This is largely because the strategies used in designing them are borrowed from the behavioural sciences, such as from the conditioning research conducted by BF Skinner and applied deliberately to game design principles as early as 2001 (Joseph Kim 2014).

Description

Premise: The game takes place in a fictional sci-fi setting, in which an alien virus has overtaken a fleet of asteroid miners. It’s fortunate, at least, that the components required to make a vaccine are found inside the asteroids themselves. However to complicate things further, the miners are part of a diplomatic network of factions — each of whom are desperate to develop the vaccine first and return to their homeworld, abandoning the rest. Players aim to manufacture, distribute, vaccinate and exfiltrate their miners first, sabotaging other players along the way.

Platform: Players can access the game from any browser and continue their game in progress. Because of its minimalistic visual representation, the game is able to be written in JavaScript from front to back and should play nicely in most modern browsers. A purpose-built mobile application would be worth investigating, especially as people are more likely to return to a game if it is an app on their home screen (and does not require typing a URL).

Gameplay: A single match takes place over the course of a week or so in real time. Most operations happen passively, with the player logging in periodically to flick switches.

At the very beginning of the chain, an asteroid can be mined for one of three resource types, with the ability to switch to mining at given intervals. Resources and personnel are then moved systematically from an asteroid to a base ship, from the base ship over conveyor belts to sorting hubs, on belts from the hubs to a factory that transforms them into refined components, and from the factory to a lab where components are mixed into a vaccine. Personnel are sent along these same routes to wait in a limited queue in the lab, where they are injected and then finally moved to a sterile ship, from which they can be exfiltrated only once all persons have made it through the course.

A few parts along this chain are expandable, with an associated resource cost. Deciding what to upgrade and how to organise the chain forms the core of the gameplay loop, as well as micromanagement of what resource is mined at any one time and when personnel are instructed to enter a place in the chain that would otherwise hold a resource.

The asynchronous multiplayer aspect comes in the form of an event queue. At certain points in a player’s game, they are given the opportunity to send a sabotage to another player. This sabotage (such as a broken belt, tripped fuse, impure resource) is sent to the other player where it sits in a queue to be dealt out at an invisibly-determined pace. In this way, players can send each other many sabotages that come down through the pipeline in a structured way. Aside from this feature, and from the meta challenge to complete the game before anybody else does, the game is essentially single-player.

Tools and Technologies

In order to host both the game and its source code, a remote version control service like GitHub will be necessary. As far as communication tools, a development team would use platforms such as Microsoft Team and Zooms to collaborate efficiently. For the website, HTML, CSS and JavaScript will be used to make a sleek, mobile-first interface.

Concerning the game itself, the front-end will be programmed in vanilla JavaScript (making it easily adaptable to the web) and a back-end technology such as Node.js would be used to handle user data, saves states and interactions. An IDE like VSCode or Notepad++ will be important for developing the game. Programs like Aseprite, Photoshop or GIMP may come in handy for the creation of simple, minimalistic art assets.

Skills Required:

There are a few skills necessary for a team to have in order to build and launch a game like this. First you would need a designer to mock up both the website and the game’s UI elements. A front-end web developer would then be necessary for creating the site to the designer’s specifications. Meanwhile, one or more programmers with some front and back experience would work to bring the game to a functional state with a barebones visual component that can be matched to the designer’s plans at a later date. Finally, someone experienced with servers would go to work hosting the site and the database at the back end of the game.

While many of these skills are out of reach for beginners, I believe it’s possible for a team of any experience to create a single-player prototype of the game that can be hosted on the internet. A prototype version of this game without a backend would not be able to persist in real-time, but could be stepped-through for demonstration of the concept.

Outcome:

If this game is successful, the outcome will be that a small core of dedicated players would interact with the site daily, while a yet more casual group logged in semi-regularly. A playerbase of this kind provides a few different opportunities for revenue, with some options better suited than others to this specific audience. The impact of this development would be that there would be an excellent factory management game that busy friends can play asynchronously from computers all around the world. It would have a low-barrier to entry and a very rewarding gameplay loop — a successful product overall.
